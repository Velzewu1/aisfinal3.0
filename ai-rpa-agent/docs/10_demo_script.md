## The Golden Path (5–7 minutes)

### ACT 1: Navigation (30 seconds)

**Demonstrates**: current-page context understanding, voice/text input

1. Start on patient list page (`http://localhost:5173`)
2. **Say or type**: _"открой карту ивановой"_
3. → System navigates to primary exam page for patient "Иванова"
4. → Timeline shows: `context_attached (page=patient_list)` → `intent_parsed (navigate)` → `dom_action_executed`
5. **Talking point**: "The system understands the current page context and routes the navigation correctly."

### ACT 2: File Upload (45 seconds)

**Demonstrates**: patient-scoped asset ingestion, file parsing

1. On primary exam page, locate the **"Файлы пациента"** section
2. **Drag and drop** the prepared `.txt` file into the upload zone
3. → File chip appears with "…" status → changes to "✓"
4. → Timeline shows: `Файл → Обработка: history.txt` → `Файл → history.txt → актив abc123…`
5. **Talking point**: "Uploaded patient files are parsed and stored as patient-scoped assets. They enrich the AI's context for this specific patient — they're never treated as generic templates."

### ACT 3: Voice-Driven Form Fill (90 seconds)

**Demonstrates**: three-channel context (utterance + PageContext + RetrievedContext), template usage, draft preview

1. **Say or type**: _"пациент жалуется на задержку речевого развития, нарушение координации, повышенный мышечный тонус"_
2. → Timeline shows full pipeline: `voice_captured → audio_preprocessed → text_transcribed → text_normalized → context_attached → intent_parsed → validation_passed → decision_made`
3. → **Draft preview card appears** with the generated field value
4. **DO NOT click approve yet.** Point out:
   - The card shows "Черновик" badge
   - Each field is labeled with its target
   - The generated text follows clinical writing conventions (from reusable templates)
5. **Talking point**: "The system never writes to the patient record without explicit doctor approval. This draft preview is the safety gate."
6. **Click "Подтвердить"** → field fills in the form
7. → Timeline shows: `dom_action_executed`

### ACT 4: Multi-Field Dictation (60 seconds)

**Demonstrates**: multiple simultaneous field fills, knowledge-enriched generation

1. **Say or type**: _"объективно: сознание ясное, мышечный тонус повышен в нижних конечностях, рефлексы оживлены. Диагноз g80"_
2. → Draft preview shows **2 fields**: `objective_findings` and `diagnosis`
3. → The generated text for objective findings follows the template structure (from seeded `tpl_primary_exam_objective`)
4. **Talking point**: "Multiple fields filled in one dictation. The AI uses our clinic's writing templates for structure, but populates with the actual patient data from the utterance."
5. **Click "Подтвердить"** → both fields fill

### ACT 5: Schedule Generation (60 seconds)

**Demonstrates**: proactive suggestion after key fields filled, schedule CP-SAT

1. → After filling complaints + objective + diagnosis, the proactive card appears:
   _"Основные поля осмотра заполнены. Сформировать расписание?"_
2. **Click "Да"** (or say _"составь расписание на 9 дней"_)
3. → Schedule card appears with assignments grid
4. → Timeline shows: `schedule_requested → schedule_generated`
5. **Click "Открыть расписание"** → navigates to schedule view
6. **Talking point**: "The system proactively suggests the next clinical step. The schedule is generated by a CP-SAT constraint solver, not hallucinated by GPT."

### ACT 6: Status Setting (30 seconds)

**Demonstrates**: set_status intent, diary page context

1. Navigate to diary: **type** _"открой дневниковую запись"_
2. **Say**: _"массаж выполнен"_
3. → Confirmation card (high-risk operation) → confirm
4. → Massage procedure card status changes to "completed"
5. **Talking point**: "Status changes are always confirmed because they're classified as high-risk operations."

## Fallback Scenarios

### Voice fails (microphone not available)
→ Use the text input field below the voice button. All commands work identically via text.
→ **Mention**: "The text input is a first-class interface, not a fallback."

### LLM timeout or error
→ Timeline shows red error event with details.
→ Retry by repeating the utterance.
→ **Mention**: "Errors are audited via Supabase event stream for production monitoring."

### Upload fails
→ File chip shows "✗" with error tooltip.
→ Common cause: file too large (>5MB) or unsupported format.

### Draft rejected
→ Click "Отклонить" → draft is discarded, no DOM mutation occurs.
→ **Mention**: "Rejection is safe — nothing was written to the host system."

---

## Key Architecture Points to Mention

1. **Layered trust model**: LLM output is untrusted. Controller validates everything with Zod schemas. Executor only receives validated action plans.
2. **Three-channel context**: Utterance + live PageContext + retrieved knowledge assets — never merged, independently validated.
3. **Patient vs. reusable assets**: Uploaded files are patient-scoped facts. Templates are reusable style guidance. The LLM knows the difference.
4. **Draft preview gate**: No generated content reaches the host DOM without explicit human confirmation.
5. **Event sourcing**: Every step is an immutable event, synced to Supabase for audit.
